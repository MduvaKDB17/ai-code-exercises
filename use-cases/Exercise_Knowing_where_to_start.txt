EXERCISE PART 3
The core entity classes are Task, TaskStatus, and TaskPriority.
Task is the main entity — it has an id, title, description, priority, status, three different timestamps (createdAt, updatedAt, completedAt), a due date,
 and a list of tags. The business logic methods on it are update(), markAsDone(), isOverdue(), addTag(), and removeTag().
TaskStatus is an enum with four values: TODO, IN_PROGRESS, REVIEW, DONE. Each one has a string value attached (like "todo", "in_progress").
TaskPriority is an enum with four values: LOW, MEDIUM, HIGH, URGENT, each mapped to an integer (1 through 4).
A term specific to this app that I noticed is "overdue" — it's not just about the due date, it's a calculated thing based on both the due date AND
 whether the task is done.


 My sketch of how the entities relate:

Questions/confusion I had:

I wasn't sure if isOverdue() only checked the due date, or if it also considered the status — turns out it checks both.
I didn't expect there to be 4 status values — I assumed there'd just be something like "not done" vs "done."
I wasn't sure why update() has a special check for tags (!isEmpty()) when none of the other fields have that — this means you can't actually clear a
task's tags through an update, since an empty list is treated as "no change."

Q1: If you call task.update(updates) where updates has an empty tags list, what happens to the original tags?

They stay the same. The check is if (updates.getTags() != null && !updates.getTags().isEmpty()) — so an empty list fails that condition and the original
 tags are left untouched.
 This means there's currently no way to clear a task's tags through update().
Q2: Can a task ever be IN_PROGRESS and overdue at the same time? What about DONE and overdue?

Yes, IN_PROGRESS and overdue can happen together — isOverdue() only excludes DONE tasks. So a task that's TODO, IN_PROGRESS, or REVIEW can be marked
 overdue if its due date has passed. But a DONE task can never be overdue, even if it was completed after its due date — the status check
  status != TaskStatus.DONE blocks that.
Q3: What's the default priority if none is specified?

MEDIUM. This comes from the shorter Task constructors — Task(title) and Task(title, description) — which both delegate to the full constructor with
 TaskPriority.MEDIUM hardcoded in.

 Task
